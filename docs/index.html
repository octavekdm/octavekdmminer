Octave KDM

A Multiple View Interactive Environment for Software Visualization



OctaveKDM: A Multiple View Environment for MATLAB and GNU/Octave using the Knowledge Discovery Metamodel (KDM)

 

The infrastructure implemented by our research group to support the analysis of MATLAB and Octave programs is available at OctaveKDM

Program comprehension plays an important role in Software Engineering. In fact, many of the software lifecycle activities depend on program comprehension. Despite the importance of MATLAB and Octave programing languages in the Engineering and Statistical communities, little attention has been paid to the conception, implementation and characterization of tools and techniques for the comprehension of programs written in these languages. Considering this scenario, this paper presents a Multiple View Interactive Environment (MVIE) called OctaveKDM tailored to support the comprehension of programs in the aforementioned languages. OctaveKDM provides a set of coordinated visual metaphors that can be adjusted in accordance with programmers needs.

OctaveKDM is designed as a web application, developed under a service-oriented architecture, extremely flexible and robust. The data extracted from source codes is stored in a NoSQL database, structured under the Knowledge Discovery Metamodel (KDM). These are the differences of this environment compared to OctMiner, since OctMiner was designed as an Eclipse plugin and the data used to load the views are extracted from the Abstract Syntaic Tree (AST) of the target program, which every time needs to be mounted.

In order to generate the KDM of a particular MATLAB/Octave program, it was necessary to develop a secondary Parser that mounts the AST of the program source code and a main Parser that, from the mounted AST, generates the corresponding KDM, as shown in Figure 1.

Instead of developing this secondary parser that generates AST from MATLAB/Octave programs, we could use one that already exists available, but only one parser was found that would meet this need without limitation. Corresponds to the parser available in the GNU Octave source code, called parse-tree. This parser was developed as a library in C language. However, to use it, some modifications would be necessary for it to contemplate the MATLAB language, in addition to Octave, for which it was conceived. The problem of this is that this parser does not have documentation and its source code is difficult to understand, as can be seen in Figure 2.

Knowledge Discovery Metamodel (KDM)

In the KDM metamodel, we identify twelve packages with one or more class diagrams describing elements aiming at representing some part of the system under analysis. These elements will be written to an XMI file to represent the target system. The Figure 3 represents the KDM package structure.

The packages are defined in layers as conveyed in Figure 3. Packages from top layers depend on the ones from lower layers. For this reason, the Core package is the main package upon which all the others provide their respective services. In the Parser implementation described here, we use elements from the five lower packages: Core, kdm, Source, Code and Action. The Figure 4 show the hierarchy of the elements used to represent the source code of Matlab/Octave programs in this work.

The Parser Implementation

To convert MATLAB/Octave Programs to Knowledge Discovery Metamodel (KDM) instances, we implemented the following libraries: ASTOctave, OctaveKDMStructure, ASTOctaveToKDM, OctaveKDMToXMI and OctaveKDMToJSON. ASTOctave includes a parser for reading a MATLAB or Octave file and generating the corresponding AST. OctaveKDMStructure represents the KDM structure of MATLAB and Octave applications. The KDM elements used to represent MATLAB/Octave programs, with their respective hierarchy, can be view in figure 4. ASTOctaveToKDM is responsible for scanning all the source files from an Octave or Matlab project and using library ASTOctave to create the AST for each, as well as creating the project's KDM metamodel. The OctaveKDMToXMI and OctaveKDMToJSON libraries scan the data stored under the project's KDM structure and respectively generate the XMI and JSON files of that structure. The figure 5 show the main parser flow and Figure 6 show the class diagram of the proposed KDM structure. This structure is implemented in the OctaveKDMStructure library.

The KDM documentation proposes that the structure representation be described in an XMI file. Nevertheless we chose to create an similar representation in the JSON file, in order to allow inclusion of the data in the NoSQL database. This will facilitate information retrieval from the stored data for possible analysis. This was the motivation for developing the OctaveKDMToJSON library.

Infrastructure to Support the Visual Analysis

The infrastructure implemented by our research group to support the analysis of MATLAB and Octave programs is available at OctaveKDM

The architecture used in the infrastructure is based on the client/server model through the use of the Service-oriented architecture(SOA). The figure 7 show the implemented architecture. The primary motivation for this architecture relates to its flexibility. The SOA architecture was created with the purpose of integrating the diverse systems within companies with low cost and little time spent. It allows new products to be developed without the need for changes in existing systems. The technologies used in the infrastructure are shown in figure 8. They comprise the Apache Tomcat web container and the CouchDB NoSQL database management system. The REST Java API was used to create the services, jQuery Ajax, and D3.js as a Java Script library to manipulate DOM objects based on JSON data. These technologies are used to implement parts B, C, D, and E of the visualization pipeline shown in figure 7.
Figure 7 - Infrastructure to Support the Visual Analysis of MATLAB/Octave Programs
Figure 8 - Technologies Used in Proposed Infrastructure

The part A of the figure 7 represents the source code repository used to fetch source codes from the MATLAB/Octave programs. The visualization environment is limited to repositories that use Git technology. Part B represents the NoSQL CouchDB database, used to store the source code information in the KDM structure. Part C represents the database Views created within the CouchDB to get the necessary information in the format to load the visual metaphors developed through the use of the D3js library. Part D represents the Server side of the visualization environment while part E represents the Client side. The Server has classes developed as services in the Java language using REST technology. These services are responsible for calling the views implemented in CouchDB and returning the result to the Client. There is also a service on the Server to import a MATLAB/Octave project from the repository into the visualization environment. It is this service that calls the ASTOctaveToKDM library to create the project's KDM, and then save it to the database. In the Client, HTML 5 pages are implemented using Java Script, CSS 3, and the Java Script D3JS library to display the available visual metaphors. Part F represents the devices that can be used by the user in the use of the visualization environment.

Proposed Strategy to Analyse MATLAB/Octave Programs

The purpose of the infrastructure proposed in figure 7 is to support the source code analysis of MATLAB and Octave programs. In this work, an exploratory study of the infrastructures was made based on the strategy of research in levels defined in figure 17.
Figure 17 - Proposed Strategy to Analyse MATLAB/Octave Programs

Choice for this exploratory study was motivated by the frequent observation, in literature reports, of the occurrence of modules of an application that concentrate a large number of functionalities and consequently a considerable (and excessive) amount of information. Modules in these situations tend to be called and/or referenced by many other modules, in many points in the system. In object orientation, this anomaly is known as God Class. Such modules are cause significant damage to the maintainability of the application and even warp the normal development process. In procedural programming, this anomaly is cause for similar problems and are likewise a threat to the modularity of the application, exacerbating coupling between modules and thus increase maintenance efforts and costs.

The analysis strategy proposed in this section aims to guide the understanding of MATLAB/Octave programs, defining step by step in an attempt to discover in these projects source code files that have similar characteristics to the God Class symptom observed in Object Oriented systems.

USING THE VISUAL APPROACH TO ANALYZE MATLAB/OCTAVE PROGRAMS

To execute the proposed exploratory study, three visual metaphors of the treemap type and two of the chord type were developed. The treemap visual metaphor is capable of representing large volumes of hierarchical data through recursively nested rectangles representing domain-relevant entities also known as real attributes. The chord view is capable of displaying a set of arcs forming a circle and a set of lines connecting each of these arcs to represent their relationships.

Ten MATLAB/Octave projects were selected based on pre-established criteria. The ten projects were imported from Github to the visualization environment and then the proposed strategy was applied to each project. Figure 18 shows the treemap visual metaphor of the matlab2tikz project. Each rectangle of this view represents the functions of source code files. All functions that are part of the same file are painted with the same color. The size of the rectangle varies according to the number of lines of code of the function.
Figure 18 - Treemap view of the MATLAB2Tikz project that represents the size of your functions and files

All rectangles are painted in the same color, and the size varies according to the amount of the file relationships. These relationships represent the coupling of the files. In phase 2, the five files with the most afferent relations and the five files with the most efferent relations were identified. The intersection between the results of these two phases of the study corresponds to candidate files possessing God Class characteristics.
